# M11 / U11-c2 簡報重構規格

## 目標

把現有 U11-c2 從「Claude Code 工具介紹課」改成:

> **Vibe Coding with Guardrails:用 Claude Code 管理 AI 小團隊完成第一個功能**

這份文件是給簡報 agent 的製作規格,不是學生講義。請依此重塑 `slides/slides/u11-c2-plan/index.tsx`。

## 為什麼要改

U11-c1 已經講過很多工具基礎:

- VSCode、Extensions、npm、cd
- Git / GitHub / remote / push / pull / branch / tag
- Claude Code 是什麼、slash commands、Modes、Plan Mode
- Codex 是什麼、核准權限、推理等級、規劃模式
- AI 動手前防呆、`git diff` 驗收

所以 U11-c2 不應再重複「裝 Claude Code、介紹模式、改一句標語」。那會變成 C1 重播。

U11-c2 應該進入第一個真正的 AI 開發 workflow:學生照老師提供的一串指令 / prompt,用 Claude Code 做出一個可見功能,並在過程中理解 vibe coding、專案守則檔、workflow、sub-agent 分工、人類審核。

## 新定位

### 課程一句話

**Vibe Coding 不是亂感覺,是用護欄把 AI 的速度變成可交付成果。**

### 學生最後要帶走

學生不需要自己發明 prompt。老師提供固定 prompt,學生照貼、照看、照驗收。

最後學生應該完成:

- 一份 `AGENTS.md`
- 一份 `CLAUDE.md`
- 一個「智慧推薦區塊」功能
- 一次 planner → implementer → reviewer 的 AI 分工流程
- 一次 `git diff` / `npm run build` / commit / push 收尾

## 本堂不做什麼

- 不再詳細教 Claude Code 安裝流程。C1 已提過 Extensions 與 Claude Code。
- 不再逐一介紹 slash commands。只在需要時使用 `/plan`、`/model`、`/compact` 等。
- 不安排 Codex 實作段。
- 不做 Claude vs Codex 對照。
- 不讓學生自由 prompt。所有 prompt 都由老師提供。
- 不新增套件。
- 不改 `package.json`。

## Codex 在本堂的位置

Codex 只在收束頁被點到,不是本堂主角。

要講清楚:

- `AGENTS.md` 是比較通用的 agent rule file。
- Claude Code 主要讀 `CLAUDE.md`。
- Codex / 其他 coding agent 可讀 `AGENTS.md`。
- 因此本堂把共用規則寫進 `AGENTS.md`,再讓 `CLAUDE.md` import 它,是為了讓同一套工作守則未來能跨工具使用。

不要做 Codex 實作頁。不要讓 Codex 重新做功能。不要做工具比較表。

## 建議總頁數與時間

目標 55-65 頁,4 小時。

觀念頁最多 12-15 頁,其餘都要綁操作、prompt、驗收、畫面變化。

建議時間:

| 時間 | 段落 | 重點 |
|---|---|---|
| 0:00-0:12 | 開場 | Vibe Coding 定義、風險、今天的任務 |
| 0:12-0:30 | 專案守則檔 | `AGENTS.md` / `CLAUDE.md` 是寫給 AI 的規則 |
| 0:30-0:45 | Workflow + sub-agent | planner / implementer / reviewer |
| 0:45-1:10 | 任務規格 | 智慧推薦區塊、資料規則、驗收標準 |
| 1:10-2:05 | 第一輪 Claude Code | Plan → 人審核 → Execute |
| 2:05-2:15 | Break | 休息 |
| 2:15-2:55 | 第二輪需求變更 | 改推薦策略,再次 Plan → Execute |
| 2:55-3:30 | Reviewer + 修偏 | reviewer 檢查 diff、超界修改、build |
| 3:30-3:50 | Git 收尾 | status / diff / build / commit / push |
| 3:50-4:00 | 收束 | 這套 workflow 未來可搬到 Codex / 其他 agent |

## 建議段落與頁面

### 開場:Vibe Coding,但要有護欄

目的:補現代名詞,但不要講太久。

建議頁面:

1. Cover  
   標題:「Vibe Coding with Guardrails」
   副標:「用 Claude Code 管理 AI 小團隊完成第一個功能」

2. What is Vibe Coding  
   定義:
   > 用自然語言描述想法,讓 AI 直接產生、修改、串接程式;人一邊看結果,一邊迭代。

3. Vibe Coding 的好處  
   - 快速看到畫面
   - 適合 prototype
   - 不熟語法也能開始
   - 想法可以快速變成產品雛形

4. Vibe Coding 的風險  
   - AI 會猜
   - AI 會多改
   - 能跑不代表好維護
   - 不看 diff 就是在賭

5. 本堂 punchline  
   > Vibe Coding 不是亂感覺,是有護欄的快速迭代。

### 專案守則檔:AGENTS.md / CLAUDE.md

目的:讓學生知道 AI 不是每次都從零理解專案;規則可以寫進 repo。

建議頁面:

6. 為什麼需要專案守則檔  
   比喻:新同事進專案,要先看團隊守則。

7. `AGENTS.md` 是什麼  
   - 給 coding agent 看的通用專案規則
   - 寫專案怎麼跑、哪些檔可改、哪些不可改、驗收指令
   - Codex / 其他 agent 可讀

8. `CLAUDE.md` 是什麼  
   - Claude Code 的專案記憶 / 指令檔
   - 可以寫 Claude Code specific rules
   - 可以用 `@AGENTS.md` 引入共用規則

9. 重要限制  
   - 這些檔案是 context,不是硬性安全鎖
   - AI 可能仍會出錯
   - 真正保險仍要靠 Plan Mode、權限、human review、git diff

10. 學生要建立的 `AGENTS.md`  
   用大字展示,不要塞滿。完整內容放成 prompt 卡。

```md
# AGENTS.md

## Project
This is a React/Vite practice project for U11.

## Allowed files for this lesson
- src/data.js
- src/App.jsx
- src/styles.css

## Do not edit
- package.json
- package-lock.json
- node_modules/
- .git/

## Workflow
1. Plan first.
2. Do not edit before approval.
3. Make the smallest change.
4. After editing, summarize changed files.
5. Run npm run build before commit.

## Git rules
- Check git status before editing.
- Use git diff after editing.
- Do not push unless the human confirms.
```

11. 學生要建立的 `CLAUDE.md`

```md
# CLAUDE.md

@AGENTS.md

## Claude Code specific
Use Plan Mode before editing.
Ask before changing files.
If the requested change touches package.json, stop and ask first.
```

12. 操作頁:建立兩個檔案  
   不是讓學生自己想內容,而是複製老師提供內容。

固定操作:

```bash
code AGENTS.md
code CLAUDE.md
```

或在 VSCode Explorer 右鍵 New File。

驗收:

- Explorer 看得到 `AGENTS.md`
- Explorer 看得到 `CLAUDE.md`
- `CLAUDE.md` 第一行附近有 `@AGENTS.md`

### Workflow + sub-agent:把 vibe 變流程

目的:把 Human-in-the-loop、Plan and Execute、sub-agent 放進同一條流程講,不要分散成長篇觀念。

建議頁面:

13. Workflow 是什麼  
   > 不是想到什麼就問什麼,而是固定照「先讀、先計畫、再改、再驗收」走。

14. 本堂固定 workflow  

```text
需求 → planner → 人審計畫 → implementer → reviewer → build → commit → push
```

15. Human-in-the-loop 放在哪  
   三個拍板點:
   - 需求拍板:今天只做智慧推薦,不做購物車、不做登入
   - 計畫拍板:AI 要改哪些檔,人先同意
   - 結果拍板:看 diff / build 後,人才 commit / push

16. Sub-agent 是什麼  
   > 把 AI 當小團隊,不同角色做不同事。

17. 三個角色  
   - planner:只讀專案、列計畫
   - implementer:照核准計畫改檔
   - reviewer:根據 `AGENTS.md` 檢查 diff

18. 注意  
   這裡不用深入 agent 架構。對學生只要講成「角色分工」。本堂可以用 prompt 讓 Claude Code 扮演不同角色即可。

### 任務規格:智慧推薦區塊

目的:功能要比「改標語」更有料,但仍然可控。

建議頁面:

19. 今天要做的功能  
   新增「智慧推薦區塊」。

20. 功能規格  
   - 從商品清單中選出一個推薦商品
   - 如果有 `featured: true` 且 `stock > 0`,優先推薦
   - 否則推薦 `rating >= 4.5` 且 `stock > 0`
   - 如果都沒有,顯示「今天沒有推薦商品」
   - 顯示商品名稱、分類、價格、推薦理由

21. 允許改的檔案  
   - `src/data.js`
   - `src/App.jsx`
   - `src/styles.css`

22. 不准改的檔案  
   - `package.json`
   - `package-lock.json`
   - `.git/`
   - `node_modules/`

23. 驗收標準  
   畫面:
   - 有「今日推薦」
   - 有商品名稱、分類、價格、推薦理由
   - featured 商品優先
   - 沒庫存不推薦
   - 無符合商品時有 fallback

   Git:
   - diff 只出現允許檔案
   - `package.json` 沒變
   - `npm run build` 通過

### 第一輪:Claude Code 做功能

目的:學生照 prompt 跑,老師帶他們看 AI 是否守規矩。

建議頁面:

24. 開始前檢查  

```bash
git status
```

畫面要看到 working tree clean,或老師帶學生先處理。

25. Planner prompt 卡  

```text
你現在先扮演 planner。

請讀:
- AGENTS.md
- CLAUDE.md
- src/data.js
- src/App.jsx
- src/styles.css

我要新增「智慧推薦區塊」。
先不要改檔。

功能規格:
1. 從商品清單中選出一個推薦商品
2. 如果有 featured: true 且 stock > 0,優先推薦它
3. 否則推薦 rating >= 4.5 且 stock > 0 的商品
4. 如果都沒有,顯示「今天沒有推薦商品」
5. 畫面要顯示商品名稱、分類、價格、推薦理由

限制:
1. 只能改 src/data.js、src/App.jsx、src/styles.css
2. 不要新增套件
3. 不要改 package.json
4. 不要重構整個專案

請輸出:
1. 會改哪些檔案
2. 每個檔案改什麼
3. 推薦邏輯會放在哪裡
4. 驗收方式
5. 可能風險
```

26. 人審計畫頁  
   用 checklist 呈現:
   - 它有沒有只改允許檔案?
   - 它有沒有要新增套件?
   - 它有沒有要改 `package.json`?
   - 它有沒有要重構?
   - 它有沒有寫驗收方式?

27. Implementer prompt 卡

```text
你現在扮演 implementer。

請照剛剛核准的計畫實作。

限制:
1. 只能修改 AGENTS.md 允許的檔案
2. 不要新增套件
3. 不要改 package.json
4. 不要重構整個專案
5. 完成後列出實際改了哪些檔案
```

28. 等 AI 修改時要看什麼  
   - 它是否開始改非允許檔案
   - 它是否要求安裝套件
   - 它是否突然大重構
   - 有問題就停止,不要接受

29. localhost 驗收  
   跑:

```bash
npm run dev
```

看:
   - 新區塊出現
   - 文案和商品資訊合理
   - 頁面沒有壞

### 第二輪:需求變更

目的:讓學生體會 vibe coding 是快速迭代,但每輪仍要守 workflow。

建議頁面:

30. 需求變更:改推薦策略  
   改成「低庫存促銷優先」。

31. Planner prompt 卡:第二輪

```text
你現在先扮演 planner。

現在要修改推薦規則。
先不要改檔。

新規則:
1. 不要優先 featured
2. stock > 0
3. stock <= 3 的商品優先
4. rating >= 4.0
5. 推薦理由要說明「庫存不多,適合促銷」

請先列出:
1. 需要改哪些檔案
2. 推薦邏輯要改哪一段
3. 可能風險
4. 驗收方式
```

32. 人審第二輪計畫  
   一樣用 checklist,重點是確認它沒有趁機改版面或改套件。

33. Execute prompt 卡:第二輪

```text
請照剛剛核准的第二輪計畫修改。

限制不變:
1. 只能改 src/data.js、src/App.jsx、src/styles.css
2. 不要新增套件
3. 不要改 package.json
4. 不要重構

完成後列出實際改了哪些檔案。
```

34. 第二輪畫面驗收  
   - 推薦理由有變
   - 低庫存商品優先
   - 畫面仍正常

### Reviewer + 修偏

目的:把 sub-agent 的 reviewer 角色落地。

建議頁面:

35. Reviewer prompt 卡

```text
你現在扮演 reviewer。

請根據 AGENTS.md 和 CLAUDE.md 檢查目前 git diff。
不要改檔。

請回報:
1. 實際改了哪些檔案
2. 是否有超出允許範圍
3. 是否有修改 package.json
4. 是否有過度重構
5. 功能是否符合第二輪需求
6. 還需要人工檢查什麼
```

36. Git diff 指令頁

```bash
git status
git diff -- src/data.js src/App.jsx src/styles.css
git diff -- package.json
```

`git diff -- package.json` 應該沒有輸出。

37. AI 超界時的救援 prompt

```text
請只保留「智慧推薦區塊」相關修改。
把所有和這個功能無關的修改還原。
不要改 package.json。
不要新增套件。
不要重構。

先列出你會還原哪些地方,等我同意後再改。
```

38. Build 驗收

```bash
npm run build
```

通過才進 Git 收尾。

### Git 收尾

目的:把「成果可交付」講清楚。

建議頁面:

39. Commit 前最後檢查

```bash
git status
git diff
npm run build
```

40. Commit

```bash
git add .
git commit -m "新增智慧推薦區塊"
```

41. Push

```bash
git remote -v
git push
```

提醒:

- `origin` 必須是學生自己的 repo。
- 看到老師 repo 就停止。

### 收束

42. 今天真正學到的不是推薦商品  
   是:
   - vibe coding
   - rule files
   - workflow
   - sub-agent roles
   - human review
   - Git 收尾

43. Codex 一頁帶過  
   標題:「為什麼我們同時寫 AGENTS.md?」
   內容:
   - Claude Code 用 `CLAUDE.md`
   - `CLAUDE.md` 可 import `AGENTS.md`
   - Codex / 其他 agent 可讀 `AGENTS.md`
   - 所以規則寫在 `AGENTS.md`,未來更容易跨工具

44. Punchline
   > 會 vibe coding 不稀奇;能讓 AI 每一輪都照流程交付,才是真本事。

45. 作業
   - 把智慧推薦區塊 push 到自己的 repo
   - 截圖:畫面 + `git log --oneline -1`
   - 貼上 reviewer 回報

## 舊 U11-c2 頁面處理建議

### 保留但改寫

- Cover:改成 Vibe Coding with Guardrails
- Roadmap:改成本文件的新 workflow
- Mode / Plan 相關頁:只保留必要提醒,不要再當主教學
- Pitfall / Checkpoint 元件:保留,但內容改成本堂新任務
- git diff / commit 頁:保留概念,換成智慧推薦區塊的 diff

### 刪除或大幅縮短

- Claude Code 安裝頁
- Claude Marketplace / 官方頁截圖大段介紹
- slash commands 大表
- 模型選擇長篇說明
- 只改標語的實作主線
- Codex 對照實作段

### 新增

- Vibe Coding 定義與風險
- `AGENTS.md` / `CLAUDE.md`
- workflow 圖
- sub-agent 三角色圖
- 智慧推薦區塊規格

## 視覺設計建議

- 延續 U11-c1 的卡片、終端機框、Checkpoint、Pitfall 風格。
- 不要做大量文字牆;prompt 可以用深色 terminal/prompt card。
- Workflow 用水平流程圖:

```text
Need → Planner → Human Gate → Implementer → Reviewer → Build → Git
```

- Sub-agent 用三張角色卡:
  - planner:腦袋 / 計畫
  - implementer:工具 / 修改
  - reviewer:放大鏡 / diff

- `AGENTS.md` / `CLAUDE.md` 用檔案卡呈現,強調「寫進 repo 的規則」。

## 防呆要求

簡報必須有以下防呆頁:

1. 開始前 `git status`
2. AI 計畫審核 checklist
3. 允許 / 禁止修改檔案
4. `git diff -- package.json` 應該無輸出
5. AI 超界救援 prompt
6. `npm run build` 通過才 commit
7. `git remote -v` 確認 origin 是學生自己的 repo 才 push

## 完成標準

簡報 agent 完成後,這堂 C2 應該看起來像:

- 不是工具導覽
- 不是改一句文字
- 不是純觀念課
- 是一堂「老師給固定 prompt,學生照流程管理 Claude Code 做出功能」的實戰課

最終學生做出的成果是智慧推薦區塊;真正學到的是可控的 vibe coding workflow。
