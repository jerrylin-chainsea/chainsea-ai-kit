好，那我們先想一個問題。

你要教一群完全沒基礎的人「AI 開發工具」。
你第一個直覺可能會是：那就介紹 VSCode、CLI、Git、Claude Code、Codex 嘛。

但這樣會有一個問題。

學生會聽到一堆工具名，然後完全不知道它們之間的關係。
就像你把鍋子、瓦斯爐、刀子、砧板、冰箱全部介紹完，但學生還是不知道今天要煮什麼。

所以這堂課要反過來設計：

> **不是從工具開始講工具，而是先讓學生看到一個固定問題，再讓每個工具自然出場。**

這也比較符合你要的李宏毅式教法：先有脈絡，不要流水帳；每一段有一個 punchline；不要直接講方法，要讓學生看到「這個方法為什麼會被想出來」。這個原則也是你給的 skill 裡明確整理出的教學方法。

# 教材的主故事

我會設計成：


工具會教，但每個工具只綁一個明確任務。

| 工具             | 這堂課怎麼教                | 學生要會什麼                                  |

| CLI / Terminal | 跑專案                   | `npm run dev`、`python run.py`           |
| Git            | 建立存檔點                 | `status`、`diff`、`add`、`commit`          |
| GitHub         | 知道遠端與公開流程概念           | commit / push / Actions / Pages 的位置關係   |
| Claude Code    | 終端機裡的 coding agent    | 讀專案、Plan、改檔、Debug                       |
| Codex          | OpenAI 的 coding agent | 讀專案、修改、review、ship code                 |
| Browser        | 驗收畫面                  | localhost、重新整理、手機尺寸                     |
| CSV / JSON     | 看資料                   | 欄位、資料列、異常值                              |
| Skills         | 固定 AI 工作 SOP          | 看懂一份簡化 Skill                            |
| MCP            | AI 接外部工具              | 知道它是工具連接，不是魔法                           |

Claude Code 官方定位就是在 terminal 裡運作的 agentic coding tool，可以理解 codebase、透過自然語言協助修改，並支援檔案編輯、執行測試與其他命令。([Claude Platform Docs][1]) Codex 則是 OpenAI 的 coding agent，官方說法是協助 write、review、ship code，也提供 app、CLI、IDE extension、web 等入口。([OpenAI Help Center][2])

所以我不會把 Claude Code / Codex 當成兩堂產品介紹。
我會這樣講：

> **Claude Code 和 Codex 是兩台不同牌子的車。今天不是比車，而是教你怎麼開車不撞牆。**


# C2：不要讓 AI 直接動手

## 課程時間

4 小時

## 這課的 punchline

> **AI 不是不能改，是不能一開始就改。**

## 教學問題

老師開場：

```text
假設你跟 AI 說：幫我改漂亮一點。
它會怎樣？

它可能改顏色。
可能改版面。
可能加動畫。
可能順便新增登入。
甚至可能重構整個專案。

你本來只是要換一個標題。
結果它像實習生第一天上班，直接把整間店重新裝潢。
```

所以方法出現了：

```text
先讀專案。
再計畫。
再修改。
```

---

## 上課內容

### 1. Vibe Coding

這裡正式講 Vibe Coding。

```text
Vibe Coding 不是讓 AI 自由發揮。
它是人說清楚意圖、限制與驗收方式，AI 負責實作，人負責檢查。
```

你的剪報裡也把 Vibe Coding 說成「你負責思考與決策，AI 負責實現與建議」，這句可以直接沿用。

---

### 2. Claude Code / Codex 的共同流程

老師講：

```text
Claude Code、Codex 看起來介面不一樣。
但你操作它們的流程要一樣。
```

固定流程：

```text
1. 叫 AI 讀專案
2. 叫 AI 說明它看到了什麼
3. 叫 AI 進 Plan Mode
4. 人確認
5. AI 修改
6. 人驗收
7. Git 看差異
```


### 3. Plan Mode：先想再動

這裡提 Plan-and-Execute。

```text
Plan-and-Execute 其實就是：
先想清楚要怎麼做，再真的做。
```

固定 Prompt：

```text
請進入 Plan Mode，不要修改檔案。

我要把首頁標語改成：
「靠海的手作選物,我的第一個 AI 小店」

限制：
1. 只修改標題文字
2. 不要改版面
3. 不要新增套件
4. 不要修改其他區塊

請先回答：
1. 你會修改哪個檔案
2. 你會修改哪一段
3. 是否會影響其他功能
4. 我要怎麼驗收
```

## 學員產出

```text
拿到 AI 的修改計畫
可以判斷 AI 是否想改太多
```

### 4. Guardrails：AI 的護欄

老師講：

```text
Guardrails 就是護欄。
不是讓 AI 更厲害。
```

```text

請不要修改不相關區塊。
請修改前先說明。
```

```text
請只修改必要部分。
完成後請告訴我：
```

# C3：資料、固定輸出、Debug / ReAct

## 課程時間

4 小時

## 這課的 punchline

> **AI 講得很順，不代表它講得對；你要看它有沒有根據。**

## 教學問題

老師開場：

```text
前面我們讓 AI 改畫面。
但真實世界不只有畫面。
還有資料。

你可能會想說：
AI 不是很聰明嗎？丟資料給它，它就會分析。

問題來了。
如果資料裡有負數銷售量呢？
如果營收大到不合理呢？
如果某天資料缺掉呢？

AI 如果沒發現，還很有自信地給建議。
這就很危險。
```

---

## 上課內容

### 1. CSV / JSON 極簡概念

老師講：

```text
CSV 像 Excel。
JSON 像一張資料卡。
```

固定 CSV：

```csv
date,product,qty,revenue
2026-07-01,A商品,30,1800
2026-07-02,A商品,-5,-300
2026-07-03,B商品,9999,999999
2026-07-04,C商品,,
```

老師問：

```text
你沒有學程式，也看得出來哪裡怪吧？

- qty 是負數
- revenue 是負數
- 數字大到不合理
- 有空值
```

這就是資料異常的直覺。

## 學員產出

```text
能指出 sales.csv 裡 3 種異常：
負數、過大數字、缺漏
```

---

### 2. 固定輸出格式

老師講：

```text
AI 如果只是寫一大段作文，後面很難使用。
所以我們要固定輸出格式。
```

固定四欄：

```text
insight：
anomalies：
risk_level：
action_items：
```

Prompt：

```text
請根據以下 sales.csv 的內容，輸出固定四欄：

insight：
anomalies：
risk_level：
action_items：

限制：
1. 如果資料有負數、過大數字、缺漏，請列在 anomalies
2. 如果資料不足，請不要硬給結論
3. 沒有資料依據的建議，請標成「不可採用」
4. risk_level 只能是 low、medium、high
```

## 學員產出

```text
一份四欄 AI 決策摘要：
insight / anomalies / risk_level / action_items
```

---

### 3. 信任規則

老師講三句就好：

```text
資料不齊，先停。
沒有依據，不採用。
高風險，找人確認。
```

這裡可以提 human-in-the-loop。

```text
Human-in-the-loop 就是：
AI 可以先幫忙判斷。
但關鍵決策要有人看過。
```

## 學員產出

```text
能說出什麼情況不能直接採用 AI 建議
```

---

### 4. Debug 與 ReAct

先製造固定錯誤，例如：

```text
python run.py 跑完後，risk_level 輸出成「嚴重」。
但我們要求只能是 low、medium、high。
```

老師講：

```text
這時候不要說：AI 壞了。
我們要讓 AI 照流程修。
```

這裡提 ReAct。

```text
ReAct = Reason + Act + Observe

Reason：先想可能原因
Act：做一個動作，例如看檔案或跑指令
Observe：看結果，例如錯誤訊息或輸出
```

Debug Prompt：

```text
目前有一個錯誤：

python run.py 的輸出中，risk_level 不是 low、medium、high，
而是輸出了其他文字。

請先不要修改檔案。

請先分析：
1. 這個問題可能是什麼意思
2. 可能原因有哪些
3. 可能相關檔案
4. 你會怎麼確認問題
5. 建議修正方式
```

允許修正：

```text
同意，請照你的分析修正。
請只修改必要檔案。

修完後請告訴我：
1. 改了哪裡
2. 為什麼這樣改
3. 我要怎麼確認修好了
```

## 學員產出

```text
1. 一次 Debug Prompt
2. 一次修錯紀錄
3. 修正後的 python run.py 輸出
```

# C3 最終產出

```text
1. 看懂 sales.csv 的異常
2. 產出固定四欄摘要
3. 建立三條信任規則
4. 用 ReAct 修一次固定錯誤
```

# C4：把流程固定下來

## 課程時間

4 小時

## 這課的 punchline

> **會用 AI 一次不稀奇；能讓 AI 每次照流程做，才是重點。**

---

## 上課內容

### 1. Context Drift

老師講：

```text
AI 改著改著，會偏離原本目標。
這叫 Context Drift。
```

例子：

```text
你只是要改標題。
它順便改顏色、改版面、改資料。
```

拉回 Prompt：

```text
請回到原本任務。
這次只處理標題修改。
不要修改其他區塊。
```

## 學員產出

```text
知道 AI 跑偏時怎麼拉回來
```

---

### 2. Prompt Debt

老師講：

```text
你一直補一堆零碎要求，AI 後面會越來越混亂。
這叫 Prompt Debt。
```

整理 Prompt：

```text
請先整理目前最新狀態，不要修改。

請列出：
1. 目前已完成哪些功能
2. 目前仍要遵守哪些限制
3. 下一步只處理什麼
4. 哪些事情不要做
```

## 學員產出

```text
一段整理上下文的 Prompt
```

---

### 3. Regression

老師講：

```text
Regression 就是改 A 壞 B。
```

### 6. Skills

老師講：

```text
如果你每次都要提醒 AI：
先讀專案、先 Plan、不要亂改、改完要驗收。

那代表這件事可以寫成 Skill。
```

Skill 白話：

```text
Skill 就是 AI 的固定 SOP。
```

固定 Skill 範本：

```md
---
name: beginner-ai-project-workflow
description: 初學者用 AI 操作 starter 專案時使用
---

你是初學者的 AI 專案助教。

當使用者要求你修改專案時，請遵守：

1. 先閱讀專案
   - 不要直接修改
   - 先說明專案用途、啟動方式、主要檔案

2. 先進入 Plan Mode
   - 重述需求
   - 說明會改哪些檔案
   - 說明修改順序
   - 說明風險
   - 提供驗收方式

3. 修改時保持最小範圍
   - 不要新增不必要功能
   - 不要新增不必要套件
   - 不要重構無關檔案
   - 優先只改指定檔案

4. 修改完成後回報
   - 改了哪些檔案
   - 每個檔案改了什麼
   - 如何啟動
   - 如何驗收

5. 涉及資料與 AI 建議時
   - 先檢查資料是否完整
   - 標出異常
   - 沒有依據不可採用
   - 高風險決策要人工確認

6. 最後提醒
   - npm run dev 或 python run.py
   - git status
   - git diff
   - 確認沒問題後再 commit
```

## 學員產出

```text
一份 beginner-ai-project-workflow.md
```

### 7. MCP

老師講：

```text
MCP 是讓 AI 接外部工具的方式。
```

不要教寫 MCP server。

只講用途：

```text
沒有 MCP：
你要自己複製資料給 AI。

有 MCP：
AI 可以在授權範圍內讀工具裡的資料。
```

例子：

| 工具           | MCP 可以做什麼       |
| ------------ | --------------- |
| GitHub       | 讀 issue、PR、錯誤紀錄 |
| Figma        | 讀設計稿            |
| Google Drive | 讀文件             |
| Jira         | 讀任務             |
| Database     | 查資料             |

安全提醒：

```text
MCP 不是越多越好。
因為它代表 AI 可以接觸外部資料。
要知道它能讀什麼、能不能寫入、會不會碰到正式資料。
```

## 學員產出

```text
能用白話說明 MCP 是什麼，以及為什麼要注意權限
```

# 我會怎麼正式編輯教材

```text
先問一個問題
  ↓

  ↓
引出工具或方法
  ↓

  ↓
```

這就是這門課的教學節奏。

# 最後我會把課的說法改成這樣

不是：

```text
我們今天要學 VSCode、CLI、Git、Claude Code、Codex。
```

而是：

```text
各位同學大家好啊，那我們今天先想一個問題。

如果有一個 AI 很會寫程式，
但它每次都不知道該改哪裡，
那你會放心把專案交給它嗎？

不會嘛。

所以今天我們不是學一堆工具名稱。
今天我們學的是：
怎麼讓 AI 先看清楚，
怎麼讓 AI 先想清楚，
怎麼讓 AI 只改該改的地方，
以及怎麼確認它沒有亂來。
```

這樣整堂課就有主線了。
工具會出現，但不是散的。
Claude Code 和 Codex 會出現，但不是產品比較。
CSV、錯誤資料、固定輸出也會出現，但它們是這份新教材自己的內容，不依賴任何後續課程素材。
