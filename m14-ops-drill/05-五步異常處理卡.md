# 05 ｜ 五步異常處理卡

> 課2 用。每張異常事件卡都跑這五步。三個判斷貫穿全程：**影響範圍多大？要不要人工接手？對外要不要說明？**

```
① 止血     先停止錯誤擴大（暫停發布/回覆/接單/通知），不是先找原因
② 定範圍   多少顧客/訂單/資料受影響？影響對外、金錢、還是內部？
③ 定層級   問題在哪一層：資料 / 流程 / 自動化 / Agent / 介面？
④ 決策     修正 or 採備援 or 人工接手？現場可用 coding agent vibe-code 小繞道/小修正（老師在旁），大型新功能才列課後
⑤ 重測+紀錄 從問題前一站重測；寫下事件、決策、溝通、人工接手
```

## 每一步怎麼用 AI（這才是 M14 的重點）

M14 不是純人工桌遊：每一步都示範「**怎麼用 AI 來做**」，學到的是用 AI 營運的能力。

| 步驟 | 用 AI 怎麼做 |
|---|---|
| ① 止血 | 請 coding agent 指出「要關哪個開關 / 註解哪一段」，比自己翻檔快 |
| ② 定範圍 | 用 M12 決策助理或 AI 讀當下資料，估算多少訂單 / 顧客受影響 |
| ③ 定層級 | 把錯誤訊息 / log 貼給 coding agent，請它判斷是資料、流程還是程式問題 |
| ④ 決策 | 用 AI 起草對外說明、整理處理選項；**人做最終決定**。必要時用 agent vibe-code 一個小繞道 |
| ⑤ 重測 | 請 agent 幫忙跑一次、讀結果確認；把已知限制請 AI 整理成一句話 |

> 原則不變：AI 幫你**更快做到**，但**判斷與最終決定在人**；AI 給的東西要能說出依據。

## 各層級的「緊急開關」止血範例（主持時可提示）

| 出問題的層 | 怎麼「止血」（先做，之後再修） |
|---|---|
| 對外網站（Astro/M13） | 把出問題的文章/連結先「註解掉或撤下」、改回安全的舊版本再 push |
| LINE OA / 社群通知 | 到後台**暫停自動回應 / 關閉該則推播**，改人工回覆 |
| 自動化流程（n8n，U8–9） | **停用該 workflow**，避免錯誤持續擴大 |
| AI 決策（M12 CrewAI） | **暫停採用 AI 建議**，改人工判斷；要求 AI 附依據才參考 |
| 資料表（U5–6） | 鎖定/標記受影響資料，先不讓下游自動使用 |

> 重點：止血是「先停止錯誤擴大」，不是「立刻修好」。修好是後面第④⑤步的事。

## 判斷小抄

- 不確定資料對不對 → **先要求確認，不要讓 AI 硬給結論**。
- AI 給建議但沒附依據 → **不採用**，要求附依據。
- 會造成「對外錯誤資訊」或「金錢/庫存錯誤」→ 最高優先處理。
- 修不完？→ 記成「已知限制」，準備 Demo 備援（錄影/截圖）。
