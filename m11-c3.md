# M11 / U11-c3 簡報重構規格

## 目標

把目前 U11-c3 從「資料觀念 + 固定四欄 + Claude/Codex 同 bug 對照」改成:

> **AI 專案整合與驗收:把營運資料接進活動頁 Dashboard**

這份文件是給簡報 agent 的製作規格,不是學生講義。請依此重塑 `slides/slides/u11-c3-data/index.tsx`。

## 為什麼要改

U11-c1 已經教過工具基本功:

- VSCode / Extensions / npm / cd
- Git / GitHub / remote / push / pull / branch / tag
- Claude Code / Codex 的基本概念、模式、核准權限
- `git diff`、`git status`、commit / push 的基本流程

U11-c2 已經升級成:

- Vibe Coding with Guardrails
- `AGENTS.md` / `CLAUDE.md`
- planner / implementer / reviewer workflow
- Human-in-the-loop
- Claude Code 做出一個可見功能

所以 U11-c3 不應再花太多時間重講工具、模式、固定四欄、同一個 bug 換 Codex 再做一次。那會變成 C1/C2 的延長版。

U11-c3 應該開始像一堂真正的專案課:

```text
資料進來
→ 先判斷資料能不能信
→ 定義程式能讀的資料合約
→ 讓 AI 把資料接進頁面
→ 跑出可見 Dashboard
→ 用 ReAct / Plan-and-execute 修錯
→ 用 git diff / build / commit / push 收尾
```

## 新定位

### 課程一句話

**AI 做出來不算完成,通過驗收才算完成。**

### 學生最後要完成

學生不需要自己設計資料集、不需要自由發明 prompt。老師提供檔案、資料、prompt,學生照流程操作。

最後學生應該完成:

- 看懂一份小型暖身資料 `sales.csv`
- 使用新增的正式資料集 `campaign_sales.csv`
- 跑 Python 產出固定格式 `report.json`
- 讓前端活動頁讀取 `report.json`
- 做出一個營運 Dashboard / 報表區塊
- 用 Claude Code 走 Plan-and-execute 完成串接
- 用 ReAct 修 2-3 個指定錯誤
- 用 reviewer prompt + `git diff` 驗收
- `npm run build` 通過後 commit / push 到自己的 repo

### 本堂不做什麼

- 不再詳細介紹 Claude Code 安裝。
- 不再詳細介紹 Codex 安裝、市集截圖、模式對照。
- 不再讓 Codex 重做同一個任務。
- 不讓學生自由設計資料分析規則。
- 不新增套件。
- 不改 `package.json`。
- 不把四欄輸出講成純觀念,必須接到程式會讀的資料合約。

## 關鍵修正:增加資料集,不是擴大 `sales.csv`

請不要把原本 5 筆 `sales.csv` 改成 50 筆。

正確做法是:

1. **保留 `sales.csv`**
   - 用途:10-15 分鐘暖身。
   - 功能:讓學生快速看懂 CSV、負數、超大值、缺值。
   - 好處:零基礎學生不會一開始就被大表嚇到。

2. **新增正式資料集 `campaign_sales.csv`**
   - 用途:本堂主實作。
   - 筆數:30-50 筆。
   - 角色:像真的活動營運資料,足夠撐起 Dashboard、資料檢查、AI 除錯。

建議欄位:

```csv
date,product,category,qty,unit_price,revenue,channel,stock,campaign
```

建議故意放入的資料問題:

- `qty` 是負數
- `revenue` 缺值
- `qty` 超大,例如 9999
- `revenue != qty * unit_price`
- `stock = 0` 但仍有銷售
- `channel` 拼字不一致,例如 `Line`, `LINE`, `line`
- 同一天同商品重複列

## 教材中的可動程式碼

這堂課一定要讓學生知道「真正會動的是哪些檔案」。

建議主線:

```text

```

### Python 端

`run.py` 要負責:

- 讀取 `campaign_sales.csv`
- 檢查資料異常
- 計算總營收
- 計算有效資料筆數
- 找出 Top product / Top channel
- 產生固定格式 `report.json`

`report.json` 建議格式:

```json
{
  "insight": "耳掛咖啡是本週營收最高商品。",
  "metrics": {
    "total_revenue": 128400,
    "valid_rows": 36,
    "anomaly_count": 5,
    "top_product": "耳掛咖啡",
    "top_channel": "LINE"
  },
  "anomalies": [
    {
      "row": 8,
      "field": "revenue",
      "problem": "revenue 與 qty * unit_price 不一致"
    }
  ],
  "risk_level": "high",
  "action_items": [
    "先人工確認第 8 列與第 17 列的營收資料。"
  ]
}
```

### 前端端

前端要負責:

- 顯示「營運 Dashboard」
- 顯示總營收、異常數、最高風險、Top product
- 顯示 anomalies 清單
- 根據 `risk_level` 顯示不同樣式
- 如果 `report.json` 缺欄位,畫面要有 fallback,不要整頁壞掉

### 段 5:ReAct 修錯

目的:把 ReAct 變成真正的除錯流程,不是只講名詞。

建議做 3 個指定 bug,每個 bug 都走:

```text
Reason → Act → Observe → Verify
```

#### Bug 1:格式錯

現象:

- `risk_level` 印出「嚴重」
- 但資料合約規定只能 `low / medium / high`

教學重點:

- 先分析,不要直接改
- 找到 `run.py` 翻譯成中文的地方
- 改完重跑 `python run.py`

#### Bug 2:計算錯

現象:

- `total_revenue` 包含異常列
- 或沒有排除 `revenue != qty * unit_price` 的列

教學重點:

- 不是所有資料都能直接加總
- AI 要先說清楚採用 / 排除哪些列
- action item 要能追到 anomalies

#### Bug 3:前端顯示錯 / regression

現象:

- Dashboard 出現了,但原本活動頁樣式壞掉
- 或 AI 改了不該改的區塊

教學重點:

- Context drift:AI 做著做著偏題
- Regression:新功能不能弄壞舊功能
- 用 `git diff` 抓超界修改

31. ReAct 解釋頁  
    - Reason:先想可能原因
    - Act:做一步,例如看檔案或跑指令
    - Observe:看結果
    - Verify:重跑測試 / build / diff

## Codex 在本堂的位置

Codex 只保留 1 頁,不要做實作段。

> 同一套驗收流程,未來可以搬到 Codex

## 可提前帶入的 C4 概念

C3 可以輕輕帶到以下概念,但不要展開成 C4:

- Context drift:AI 做著做著偏離任務。
- Prompt debt:一直補口頭規則,不如整理進 `AGENTS.md` / `CLAUDE.md`。
- Regression:新功能出現,舊功能壞掉。

C4 仍保留更進階的:

- Skills
- MCP
- 自動化 workflow
- 更完整的 agent 工具鏈

## 舊 U11-c3 頁面處理建議

### 保留但改寫

- CSV / JSON 解釋:保留,縮短。
- `sales.csv` 異常資料:保留,當暖身。
- 固定四欄:改成資料合約 `report.json`。
- ReAct:保留,改成 2-3 個真 bug。
- `git diff`:保留,升級成 reviewer 驗收。
- Human-in-the-loop:保留,放到計畫審核、結果審核、commit/push 前。

### 刪除或大幅縮短

- 一欄一欄講 `insight / anomalies / risk_level / action_items` 的長段落。
- Codex Marketplace / install / IDE features 截圖。
- Claude vs Codex 表格。
- 同一個 `risk_level` bug 用 Codex 再修一次。

## 防呆要求

簡報必須有以下防呆頁:

1. `git status` 開始前檢查。
2. `git remote -v` 確認 `origin` 是學生自己的 repo。
3. AI 計畫審核 checklist。
4. 允許 / 禁止修改檔案。
5. `python run.py` 驗證資料輸出。
6. `npm run build` 驗證前端。
7. `git diff -- package.json` 應該沒有輸出。
8. AI 超界救援 prompt。
9. commit 前看 diff。
10. push 前再次確認 remote。

## 完成標準

簡報 agent 完成後,這堂 C3 應該看起來像:

- 不是資料名詞課。
- 不是固定四欄作文課。
- 不是 Codex 對照課。
- 是一堂「資料進來 → AI 接功能 → 人類驗收 → 修錯 → 交付」的專案課。

最終學生做出的成果是營運 Dashboard;真正學到的是:

- 資料不能盲信
- AI 輸出要有合約
- 功能要能驗收
- 除錯要走 ReAct
- AI 修改要看 diff
- 通過 build 才能 commit / push
